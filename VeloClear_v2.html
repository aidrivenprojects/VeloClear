Phase 5 › RACI Matrix
RACI Matrix
R = Responsible (does the work) · A = Accountable (ONE person — owns the outcome, cannot be shared) · C = Consulted (provides input before decisions) · I = Informed (receives updates after decisions). Every row must have exactly one A — missing or duplicate A is the primary failure mode in distributed teams.
💡
Why RACI resolves conflict: At Apex, a 3-day dev/BA email dispute was resolved in 10 minutes by showing the RACI to both parties. When both can see exactly whose decision it is, there is no argument. Structure is faster than mediation.

Phase 7 › Velocity Tracker
Velocity Tracker
Story points completed per sprint. Use the average to forecast delivery dates — NOT as a performance target. Forcing velocity up creates quality debt within 2–3 sprints. Velocity is a forecasting tool.
🎯
Velocity Chart — Last 8 Sprints
Formula: Forecast sprints = Remaining backlog points ÷ Average velocity
Avg velocity = sum of last N sprint points ÷ N · Forecast = Remaining ÷ Avg

X-axis: S1–S8 = Sprint 1 to Sprint 8 (each = one 2-week delivery cycle) · Y-axis: Story points completed that sprint · Dashed line: Rolling average — use this number for forecasting · Declining bars: Risk signal — investigate at Sprint 3, not Sprint 5

Three columns: What went well · What to improve · Actions with owners and due dates. Every retro generates action items that carry forward to the next sprint. A retro every 2 weeks = 12 improvement cycles per project year.
💡
How to run a retro: Give everyone 5 minutes to add cards to each column. Group similar items. Vote on the top 2–3 items to improve. Assign one action item per improvement with a named owner. Keep it to 45 minutes. The goal is one concrete improvement per sprint — not a list of complaints.

Delivery Health Dashboard
All five delivery metrics in one view: Velocity · Burn-Down · Burn-Up · Throughput · Predictability. The JD names all five explicitly. Every chart shows the formula used and what the axis labels mean.
📈
Burn-Up Chart — Release Progress
Done line vs Total Scope · Rising scope line = informal addition → raise a CR
Scope line flat = discipline · Scope line rising = trigger a Change Request

X-axis: S1–S8 = Sprint 1 to 8 (the full release timeline, not a single sprint) · Y-axis: Story points — Done line rises from zero, Scope line is the ceiling · Gap at S8: Remaining work — extend or descope via formal CR

🌊
Throughput — Items Completed per Sprint
Flow metric · Independent of story size · Different from velocity
Velocity = story points (effort) · Throughput = items completed (flow) · Both needed

X-axis: S1–S5 = Sprint 1 to 5 · Y-axis: Number of work items completed (not story points) · Low throughput + high velocity: Large items stuck in progress — break them down

🎯
Predictability — PI Objective Attainment %
Formula: Delivered PI Objectives ÷ Committed PI Objectives × 100 · Target ≥80% · Below 70% = RCA
Predictability % = Delivered ÷ Committed × 100 · Target ≥80% · Below 70% triggers Root Cause Analysis

X-axis: PI 1–PI 4 = Programme Increments (each 8–12 weeks in SAFe) · Y-axis: % of PI Objectives delivered vs what was committed at PI Planning · Green line at 80%: Minimum healthy target · Red line at 70%: Danger threshold — below this triggers Root Cause Analysis · Above 100%: Under-commitment (sandbagging)

🎯 Add Risk to RAID Log
A risk is a potential future problem. The Trigger field is the most important — it is the specific observable event that forces your mitigation plan to activate automatically.
Risk Details
Example: "Third-party API vendor may fail to deliver the integration module on schedule, causing mobile app launch delay." Be specific — vague risks cannot be mitigated.
Technical = tech failures · Commercial = vendor/budget · People = resource/skills · External = regulatory/market
The person responsible for watching this risk and activating mitigation when the trigger fires. Only one owner — shared ownership means no ownership.
Scoring — Formula: Score = Probability × Impact
3
1=Rare · 2=Unlikely · 3=Possible · 4=Likely · 5=Almost certain
3
1=Negligible · 3=Significant delay/cost · 5=Project-ending or irreversible
9
1–4=Low (accept) · 5–14=Medium (monitor) · 15–25=High (must mitigate)
Trigger & Response — Most important section
Good example: "Vendor misses their Sprint 2 milestone delivery (due 20 Jan)" — this is specific and observable. Bad example: "Vendor seems slow" — this is vague and unactionable. When this trigger fires, mitigation activates immediately.
Mitigate=Reduce P or I · Transfer=Insurance/vendor liability · Avoid=Remove the cause · Accept=Acknowledge and monitor
Example: "Build internal API wrapper in parallel from Sprint 1. Maintain as fallback. This reduces Impact from 5 to 2 — project delivers 3 days late instead of 3 weeks late."
Example: "Activate internal wrapper immediately. Brief steering within 24 hours. Escalate to vendor account director (not project contact). Revise delivery date by 3 days."

📝 Add User Story
A user story describes a feature from the end user's perspective. Format: "As a [user], I want [goal], so that [reason]." Every story needs acceptance criteria — the specific conditions that define Done.
Story Details
Example: "As a branch manager, I want to receive push notifications for urgent client alerts, so that I can respond within 30 minutes even when away from my desk."
The larger feature theme this story belongs to. Epics group related stories together.
Must Have = release fails without it · Should Have = important but workaround exists · Could Have = nice to have · Won't Have = explicitly out of scope this release
Relative effort agreed by the team — not hours. If a story is 13 points, split it into smaller stories first.
How much value is lost every sprint this is NOT delivered? 10 = critical path, revenue at risk. Used for WSJF prioritisation.
Acceptance Criteria — What does Done look like?
Example: "Given a branch manager is logged in, When a client alert is triggered, Then a push notification is received within 60 seconds with client name and alert type. Given the app is in background, When an alert fires, Then the notification still appears on the lock screen." These criteria define when the story can be marked Done.
